# 在 Cursor 中创建 Skill

本 Skill 指导你创建有效的 Cursor Agent Skill。Skill 是教 Agent 如何执行特定任务的 Markdown 文件：使用团队标准审查 PR、以首选格式生成提交信息、查询数据库架构，或任何专门的工作流程。

## 开始之前：收集需求

在创建 Skill 之前，从用户那里收集以下基本信息：

1. **目的和范围**：此 Skill 应帮助完成什么特定任务或工作流程？
2. **目标位置**：应该是个人 Skill（`~/.cursor/skills/`）还是项目 Skill（`.cursor/skills/`）？
3. **触发场景**：Agent 应在何时自动应用此 Skill？
4. **关键领域知识**：Agent 需要什么它还不知道的专业信息？
5. **输出格式偏好**：是否有特定的模板、格式或样式要求？
6. **现有模式**：是否有要遵循的现有示例或约定？

### 从上下文推断

如果你有之前的对话上下文，可以根据讨论的内容推断 Skill。你可以基于对话中出现的工作流程、模式或领域知识创建 Skill。

### 收集额外信息

如果需要澄清，请使用 AskQuestion 工具（如果可用）：

```
AskQuestion 使用示例：
- "这个 Skill 应该存储在哪里？" 选项如 ["个人 (~/.cursor/skills/)", "项目 (.cursor/skills/)"]
- "这个 Skill 应该包含可执行脚本吗？" 选项如 ["是", "否"]
```

如果 AskQuestion 工具不可用，请以对话方式询问这些问题。

## Skill 文件结构

### 目录布局

Skill 以包含 `SKILL.md` 文件的目录形式存储：

```
skill-name/
├── SKILL.md              # 必需 - 主要说明
├── reference.md          # 可选 - 详细文档
├── examples.md           # 可选 - 使用示例
└── scripts/              # 可选 - 实用脚本
    ├── validate.py
    └── helper.sh
```

### 存储位置

| 类型 | 路径 | 范围 |
|------|------|-------|
| 个人 | `~/.cursor/skills/skill-name/` | 在你所有项目中可用 |
| 项目 | `.cursor/skills/skill-name/` | 与使用此仓库的任何人共享 |

**重要**：切勿在 `~/.cursor/skills-cursor/` 中创建 Skill。此目录保留给 Cursor 的内部内置 Skill，由系统自动管理。

### SKILL.md 结构

每个 Skill 都需要一个带有 YAML frontmatter 和 Markdown 正文的 `SKILL.md` 文件：

```markdown
---
name: your-skill-name
description: 简要描述此 Skill 的作用和使用时机
---

# 你的 Skill 名称

## 说明
清晰、分步的 Agent 指导。

## 示例
使用此 Skill 的具体示例。
```

### 必需的元数据字段

| 字段 | 要求 | 用途 |
|-------|--------------|---------|
| `name` | 最多 64 个字符，仅小写字母/数字/连字符 | Skill 的唯一标识符 |
| `description` | 最多 1024 个字符，非空 | 帮助 Agent 决定何时应用 Skill |

## 编写有效的描述

描述对于 Skill 发现**至关重要**。Agent 使用它来决定何时应用你的 Skill。

### 描述最佳实践

1. **使用第三人称**（描述被注入系统提示中）：
   - ✅ 好："处理 Excel 文件并生成报告"
   - ❌ 避免："我可以帮你处理 Excel 文件"
   - ❌ 避免："你可以使用这个来处理 Excel 文件"

2. **具体并包含触发词**：
   - ✅ 好："从 PDF 文件中提取文本和表格、填写表单、合并文档。在处理 PDF 文件或用户提及 PDF、表单或文档提取时使用。"
   - ❌ 模糊："帮助处理文档"

3. **包含 WHAT 和 WHEN**：
   - WHAT：Skill 做什么（具体能力）
   - WHEN：Agent 应该何时使用它（触发场景）

### 描述示例

```yaml
# PDF 处理
description: 从 PDF 文件中提取文本和表格、填写表单、合并文档。在处理 PDF 文件或用户提及 PDF、表单或文档提取时使用。

# Excel 分析
description: 分析 Excel 电子表格、创建数据透视表、生成图表。在分析 Excel 文件、电子表格、表格数据或 .xlsx 文件时使用。

# Git 提交助手
description: 通过分析 git diff 生成描述性提交信息。在用户请求帮助编写提交信息或审查暂存更改时使用。

# 代码审查
description: 按照团队标准审查代码质量、安全性和最佳实践。在审查拉取请求、代码更改或用户请求代码审查时使用。
```

## 核心编写原则

### 1. 简洁是关键

上下文窗口与对话历史、其他 Skill 和请求共享。每个 token 都在争夺空间。

**默认假设**：Agent 已经非常聪明。只添加 Agent 还没有的上下文。

质疑每条信息：
- "Agent 真的需要这个解释吗？"
- "我可以假设 Agent 知道这个吗？"
- "这段文字值得它的 token 成本吗？"

**好（简洁）**：
```markdown
## 提取 PDF 文本

使用 pdfplumber 进行文本提取：

\`\`\`python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
\`\`\`
```

**坏（冗长）**：
```markdown
## 提取 PDF 文本

PDF（便携式文档格式）文件是一种常见的文件格式，包含文本、图像和其他内容。要从 PDF 中提取文本，你需要使用一个库。有许多可用于 PDF 处理的库，但我们推荐 pdfplumber，因为它易于使用且能很好地处理大多数情况...
```

### 2. 保持 SKILL.md 在 500 行以内

为获得最佳性能，主要的 SKILL.md 文件应该简洁。对详细内容使用渐进式披露。

### 3. 渐进式披露

将基本信息放在 SKILL.md 中；仅当 Agent 需要时才将详细的参考材料放在单独的文件中。

```markdown
# PDF 处理

## 快速开始
[在此放置基本说明]

## 其他资源
- 有关完整的 API 详情，请参阅 [reference.md](reference.md)
- 有关使用示例，请参阅 [examples.md](examples.md)
```

**保持引用只有一层深度** - 直接从 SKILL.md 链接到参考文件。深度嵌套的引用可能导致部分读取。

### 4. 设置适当的自由度

根据任务的脆弱性匹配特异性：

| 自由度级别 | 何时使用 | 示例 |
|---------------|-------------|---------|
| **高**（文本说明） | 多种有效方法，取决于上下文 | 代码审查指南 |
| **中**（伪代码/模板） | 有首选模式但允许变化 | 报告生成 |
| **低**（特定脚本） | 脆弱操作，一致性至关重要 | 数据库迁移 |

## 常见模式

### 模板模式

提供输出格式模板：

```markdown
## 报告结构

使用此模板：

\`\`\`markdown
# [分析标题]

## 执行摘要
[关键发现的一段概述]

## 关键发现
- 有支持数据的发现 1
- 有支持数据的发现 2

## 建议
1. 具体的可操作建议
2. 具体的可操作建议

```

### 示例模式

对于输出质量取决于看到示例的 Skill：

```markdown
## 提交信息格式

**示例 1：**
输入：添加了使用 JWT 令牌的用户认证
输出：
\`\`\`
feat(auth): 实现基于 JWT 的认证

添加登录端点和令牌验证中间件
\`\`\`

**示例 2：**
输入：修复了日期显示不正确的错误
输出：
\`\`\`
fix(reports): 修正时区转换中的日期格式

在报告生成中始终使用 UTC 时间戳
\`\`\`
```

### 工作流程模式

将复杂操作分解为清晰的步骤并附带检查清单：

```markdown
## 表单填写工作流程

复制此检查清单并跟踪进度：

\`\`\`

- [ ] 第 1 步：分析表单
- [ ] 第 2 步：创建字段映射
- [ ] 第 3 步：验证映射
- [ ] 第 4 步：填写表单
- [ ] 第 5 步：验证输出
\`\`\`

**第 1 步：分析表单**
运行：`python scripts/analyze_form.py input.pdf`
...
```

### 条件工作流程模式

通过决策点进行指导：

```markdown
## 文档修改工作流程

1. 确定修改类型：

   **创建新内容？** → 遵循下面的"创建工作流程"
   **编辑现有内容？** → 遵循下面的"编辑工作流程"

2. 创建工作流程：
   - 使用 docx-js 库
   - 从头开始构建文档
   ...
```

### 反馈循环模式

对于质量关键型任务，实施验证循环：

```markdown
## 文档编辑流程

1. 进行编辑
2. **立即验证**：`python scripts/validate.py output/`
3. 如果验证失败：
   - 查看错误信息
   - 修复问题
   - 再次运行验证
4. **仅在验证通过后才继续**
```

## 实用脚本

预制的脚本比生成的代码具有优势：
- 比生成的代码更可靠
- 节省 token（代码不在上下文中）
- 节省时间（无需代码生成）
- 确保跨使用的一致性

```markdown
## 实用脚本

**analyze_form.py**：从 PDF 中提取所有表单字段
\`\`\`bash
python scripts/analyze_form.py input.pdf > fields.json
\`\`\`

**validate.py**：检查错误
\`\`\`bash
python scripts/validate.py fields.json
# 返回："OK" 或列出冲突
\`\`\`
```

明确说明 Agent 应该**执行**脚本（最常见）还是**读取**它作为参考。

---

## 应避免的反模式

### 1. Windows 风格路径
- ✅ 使用：`scripts/helper.py`
- ❌ 避免：`scripts\helper.py`

### 2. 选项太多
```markdown
# 不好 - 令人困惑
"你可以使用 pypdf，或 pdfplumber，或 PyMuPDF，或..."

# 好 - 提供带有逃生口的默认值
"使用 pdfplumber 进行文本提取。
对于需要 OCR 的扫描 PDF，请改用 pdf2image 和 pytesseract。"
```

### 3. 时间敏感信息
```markdown
# 不好 - 会过时
"如果你在 2025 年 8 月之前做这个，请使用旧 API。"

# 好 - 使用"旧模式"部分
## 当前方法
使用 v2 API 端点。

## 旧模式（已弃用）
<details>
<summary>旧版 v1 API</summary>
...
</details>
```

### 4. 术语不一致

选择一个术语并在整个过程中使用它：
- ✅ 始终使用"API 端点"（不要混用"URL"、"路由"、"路径"）
- ✅ 始终使用"字段"（不要混用"框"、"元素"、"控件"）

### 5. 模糊的 Skill 名称
- ✅ 好：`processing-pdfs`、`analyzing-spreadsheets`
- ❌ 避免：`helper`、`utils`、`tools`

## Skill 创建工作流程

在帮助用户创建 Skill 时，请遵循以下流程：

### 第 1 阶段：发现

收集有关以下信息：
1. Skill 的目的和主要用例
2. 存储位置（个人 vs 项目）
3. 触发场景
4. 任何特定要求或限制
5. 要遵循的现有示例或模式

如果你有权访问 AskQuestion 工具，请用它进行高效的结构化收集。否则，以对话方式询问。

### 第 2 阶段：设计

1. 起草 Skill 名称（小写、连字符、最多 64 个字符）
2. 编写具体的第三人称描述
3. 列出所需的主要部分
4. 确定是否需要支持文件或脚本

### 第 3 阶段：实现

1. 创建目录结构
2. 使用 frontmatter 编写 SKILL.md 文件
3. 创建任何支持参考文件
4. 如果需要，创建任何实用脚本

### 第 4 阶段：验证

1. 验证 SKILL.md 是否在 500 行以内
2. 检查描述是否具体并包含触发词
3. 确保整个过程中术语一致
4. 验证所有文件引用只有一层深度
5. 测试 Skill 是否可以被发现和应用

**SKILL.md：**
```markdown
---
name: code-review
description: 按照团队标准审查代码质量、安全性和可维护性。在审查拉取请求、检查代码更改或用户请求代码审查时使用。

# 代码审查

## 快速开始

审查代码时：

1. 检查正确性和潜在错误
2. 验证安全最佳实践
3. 评估代码可读性和可维护性
4. 确保测试充分

## 审查检查清单

- [ ] 逻辑正确并处理边缘情况
- [ ] 没有安全漏洞（SQL 注入、XSS 等）
- [ ] 代码遵循项目风格约定
- [ ] 函数大小和重点适当
- [ ] 错误处理全面
- [ ] 测试覆盖更改

## 提供反馈

将反馈格式化为：
- 🔴 **关键**：必须在合并前修复
- 🟡 **建议**：考虑改进
- 🟢 **锦上添花**：可选增强

## 其他资源

- 有关详细的编码标准，请参阅 [STANDARDS.md](STANDARDS.md)
- 有关示例审查，请参阅 [examples.md](examples.md)
```

## 总结检查清单

在最终确定 Skill 之前，请验证：

### 核心质量
- [ ] 描述具体并包含关键词
- [ ] 描述包含 WHAT 和 WHEN
- [ ] 使用第三人称编写
- [ ] SKILL.md 正文在 500 行以内
- [ ] 整个过程中术语一致
- [ ] 示例具体而非抽象

### 结构
- [ ] 文件引用只有一层深度
- [ ] 适当使用渐进式披露
- [ ] 工作流程有清晰的步骤
- [ ] 没有时间敏感信息

### 如果包含脚本
- [ ] 脚本解决问题而不是回避
- [ ] 必需的包已记录
- [ ] 错误处理明确且有帮助
- [ ] 没有 Windows 风格路径
